確定設計原則 (Charter) 3 点を実装で達成:

  1. chunk を保持しない: 観測は size と ts のみ。Buffer 本体は pipe を通じて即座に下流へ。

2. 確定設計原則 (Charter)

2.1 三原則の確定文

#原則意味違反時に起きる現象
① 制御コードはそのまま流す ANSI escape / 改行 / CR / BS / BEL / バイナリ混在 / 0x03 等の制御バイトを一切解釈・改変せず byte stream として透過 chunk 非保持原則の破綻、TTY 実装に踏み込み、無限 state 化
② シグナルは中継する SIGINT/SIGTERM/SIGHUP/SIGPIPE/SIGWINCH を親で握りつぶさず子へ伝播 Ctrl+C が効かない、zombie 化、CI/shell の期待動作崩壊

2.5 原則の不変性

この三原則は 機能追加で書き換えない / バグ修正で緩めない / 性能最適化で崩さない。 緩めたい誘惑が出たときは、そもそも nproxy の責務外である可能性が高い。 新機能はまず 外側 (outer) に置けないかを検討する。

3. Phase1: 調査結果（V8/libuv ＝ stream/IO 実装）

2.1 highWaterMark 既定値（lib/internal/streams/state.js）

let defaultHighWaterMarkBytes =
  process.platform === 'win32' ? 16 * 1024 : 64 * 1024;
let defaultHighWaterMarkObjectMode = 16;

Windows は 16 KB / 他は 64 KB。Node は read(n) で hwm 超過要求を受けると computeNewHighWaterMark が 最大 1 GiB まで動的に拡張する。 よって hwm を「上限」と信じてはいけない。

2.2 pipe の本質（src/stream_pipe.cc）

JS API readable.pipe(writable) の実体は C++ の StreamPipe クラス。 書き込みが非同期キューに積まれた瞬間、ソース側で ReadStop() が呼ばれ libuv の poll が止まる。 これにより chunk はカーネルの pipe バッファに留まり、Node ヒープには載らない。 これが nproxy 設計の根拠である。 

// stream_pipe.cc から抜粋
StreamWriteResult res = sink()->Write(&buffer, 1);
pending_writes_++;
if (!res.async) {
  writable_listener_.OnStreamAfterWrite(nullptr, res.err);
} else {
  is_reading_ = false;
  res.wrap->SetBackingStore(std::move(bs));
  if (source() != nullptr)
    source()->ReadStop();   // ← OS レベルでの読み込み停止
}

6. Phase5: プロトタイプ実装

本流は3行で終わる。それ以外は観測と policy 適用のためのコード。 

// 観測フック（観測のみ。流れは pipe が握る）
wrapWriteForObserve(child.stdin, 'in');
wrapWriteForObserve(process.stdout, 'out');
wrapWriteForObserve(process.stderr, 'err');

// 本流 pipe（C++ 層 backpressure に任せる）
process.stdin.pipe(child.stdin);
child.stdout.pipe(process.stdout);
child.stderr.pipe(process.stderr);

書き込みフックは 戻り値（false=backpressure）を維持して伝搬 し、観測はメタ情報の追記のみ。 chunk 本体への参照を残さないので GC が即座に回収できる。

class RingMetaBuffer {

  resize(newSize) { /* 末尾 newSize 件だけ復元 */ }

}

8. Phase7: 評価と結論

7.1 chunk をどう扱っているか

chunk 本体（Buffer）は 1秒も保持しない。wrapWriteForObserve は chunk.length と kind をメタ化するだけで、参照を別変数へコピーしない。 NORMAL 状態でのみ chunk.slice(0,16).toString('hex') という32文字以下の string を作るが、これも 16B ごとに使い捨てる短寿命オブジェクトであり、世代別 GC で回収される。

7.2 OOM をどこで防いでいるか

  1. C++ 層 backpressure: pipe() 経由で Node が自動で ReadStop する。 これが第一防衛線で、本流データはカーネルバッファ（プロセス外）に留まる。
  2. chunk を保持しない設計: ヒープに乗る期間が write メソッド呼び出しの間だけ。
  3. ring の縮退: メタ情報の累積も hwm が PRESSURE 閾値で打ち止まる。
  4. data イベント不使用: flowing-mode に固定する経路を完全に閉じている。

7.3 stdin / stdout / file I/O を同一思想で扱う

すべて「chunk = Buffer の Stream」として一括して捉え、差別化しないのがポイント。

系統nproxy の処理
stdin (TTY/pipe/redirect)process.stdin.pipe(child.stdin) 1 行
stdoutchild.stdout.pipe(process.stdout) 1 行
stderrchild.stderr.pipe(process.stderr) 1 行
子の fs I/O直接フックしない。流量と memoryUsage で間接観測

fs I/O が間接的に親へ波及する経路は「子が読んだものを stdout に吐く」だけなので、 stdout の Observer がそれを捕捉する。fs を直接ラップする必要はないと結論できる。

7.4 設計の最重要発見

Node.js の pipe() は単なる JS のメソッドではなく、C++ 層で OS poll を ON/OFF する OS レベルの backpressure 機構 である。 nproxy が一番すべきことは、これを邪魔しないこと。

要件にある「拒否しない／止めない／覚えない」は、JS 側で能動的にやることではなく、 能動的に何もしないことで自然に達成される性質である。観測と policy 切替は あくまで「副作用の縮退」であり、本流には影響しない。

7.5 残課題（プロダクション化への道のり）

  • worker_threads 内のヒープは memoryUsage に出ない。worker.resourceLimits で個別取得が必要
  • Windows でのシグナル中継: SIGTERM/SIGINT の挙動が POSIX と違う点の調整
  • 子プロセスが二段で spawn する場合のシグナル伝播（プロセスグループ）
  • stderr を tee する際の順序保証（OS pipe バッファの非同期性）

9.2 重要な観察

  • heap がデータサイズに対して完全フラット (10倍データでも 2.5MB → 2.6MB)。chunk が JS ヒープにほぼ滞在していない直接的な証拠。
  • RSS は時間に追従するがサイズには依らない。OS の pipe バッファ + ファイル書き込みキャッシュの可視化であり、設計の問題ではない。
  • スループットは規模が大きいほど上がる (576 → 837 → 1037 MB/s)。初期化コストが償却されるだけで、ランタイム自体が天井になっていない。
  • 状態遷移ゼロ。PRESSURE 閾値 80MB に対し heap は終始 2〜3MB。policy が走らなかったのは「走らせる必要がなかった」から。

9.3 数字の解釈

「設計上、Node が天井になる場面はもう設計の外側にある」
（100GB ETL、数百本の同時 pipe、splice/sendfile 必須要件、pty 完全制御等）

意図的に nproxy を壊そうとすると、必ず以下のどれかに該当する:

  1. 設計原則を破る（data イベントで chunk を保持） — 即 OOM。設計の問題でランタイムの問題ではない。
  2. 子プロセス側で巨大 Buffer を確保 — child が落ちる。nproxy 親は無傷。
  3. nproxy を 100 個同時起動 — OS の fd / pipe バッファ枯渇。ここで初めてランタイム差し替えが意味を持ち始める。
  4. ネットワーク bond の TCP window 律速 — Node でも対応可能だがゼロコピーが欲しくなる。

10. Phase9: Rust MVP 方針

Rust 実装の意図は 「Node では足りないから乗り換える」ではない。 「同じ思想を最下層で再証明し、機能を拡張する」フェーズである。

11. 派生する技術原則（言語横断アンカー）

このプロジェクトの根幹。Node 実装でも、Rust / Go / Zig 実装でも、この 3 点が動かなければ実装失敗。
#原則言い換え違反したときに起きる現象
① chunk 非保持
(No Chunk Retention)		 chunk を passthrough する瞬間以外、メモリに留めない。観測は size/ts/kind のメタのみ OOM（即時 / 緩慢）
② backpressure 委譲
(Delegate to OS-level)		 自前でバッファを積まない。OS poll の「読まない→詰まる→上流 blocking」に乗る flowing-mode 固定で機構が破壊。OOM か stall
③ policy は副作用の縮退のみ
(Policy Reduces Side-effects Only)		 本流は絶対に止めない／拒否しない。縮退するのは観測解像度・ring・ログだけ 「拒否しない／止めない」要件違反。プロキシの存在意義が消える

この 3 点を後続実装の 絶対要件 (must) として固定する。 README.md にも同内容が記載され、コードレビュー時のチェックリストとなる。

12. Phase10: ランタイム Text I/O 層

主題: Byte chunk を string に decode しても、設計原則を保てるか。
追加要件: Node のメモリ状況に応じて Text 処理を自動 ON/OFF できるか。

12.1 実装統合

nproxy.js に統合。CLI: 

node nproxy.js [--text=MODE] [--text-log=PATH] app.js [args...]
# MODE: off | passthrough | transform | tee
MODE処理内容ストリーム経路
offbyte 直結 (string 化なし)input → finalSink
passthroughStringDecoder で UTF-8 decode → 再出力input → TextDecode → finalSink
transformdecode + 行頭にタイムスタンプ/行番号input → TextDecode → LineTransform → finalSink
teedecode + ターミナル + ログファイル両方input → TextDecode → TeeTransform → finalSink

12.2 メモリ自動連動 (policy → text mode)

Policy statetextRequested='transform' の場合の実適用
NORMALtransform (要求モード)
PRESSUREpassthrough に自動縮退 (重い transform は停止)
CRITICALoff に強制 (text 処理完全停止)

これは設計原則3「policy は副作用の縮退のみ」の Text 層への拡張。 text 加工 (decode/行解析) を「重い副作用」と再定義し、policy で動的に縮退する。

10GB を string にdecode しながら通しても heap は 19MB 以下に張り付く。 StringDecoder の保留が C++ binding 内の固定サイズ (kSize) で完結するため、データサイズに対して 完全フラット。forcedFlush=0 (改行なしの巨大行による強制 flush も発生せず)。

12.7 chunk 境界 UTF-8 破壊検知 (T5)

app_text_boundary.js で 「あ」(E3 81 82) を 1〜13 バイトの不揃いchunkに切って 1万回連続書き込み。 chunk境界が3バイト文字の途中で切れることを保証。

結果: 30,000バイトすべて E3 81 82 の繰り返し。byteIntegrity=true。 StringDecoder の保留機構が完全に機能している証拠。

12.9 結論

思想は維持された。Text I/O は Byte I/O と同じ宇宙にあり、policy は両層を貫いて作動する。
  • 10GB を string decode しても heap 19MB 以下
  • chunk 境界の UTF-8 破壊なし (forcedFlush=0)
  • メモリ圧力で transform → passthrough に自律縮退
  • byte 層も同じく動作維持

Phase 1〜9 で築いた byte 層設計が、より重い Text I/O 領域でも壊れず通用することの実証。 Rust MVP の text 層は tokio_util::codec + StringDecoder相当 (Encoder/Decoder) で 同じ思想を写像できる。
