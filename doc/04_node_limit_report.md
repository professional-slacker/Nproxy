# Node ランタイム限界レポート

## 1. 目的

> 「設計の破綻点」ではなく「Node ランタイムの天井」を測る。

Phase6 までで、6 ケース PASS と 1GB 透過まで確認した。
ここでは **5GB を本番ライン**として、設計が Node ランタイム上でどこまで耐えうるか、
さらに参考として 1GB / 10GB の挙動を記録する。

実行コマンド (5GB の場合):

```
node run_limit_test.js 5120
```

---

## 2. 観測指標

| 指標 | 取得元 | 目的 |
|---|---|---|
| 出力ファイルサイズ推移 | runner 側 5秒tick | 進捗とstall検知 |
| heapUsed / heapTotal | nproxy debug log の TICK 行 | JSヒープがフラットか |
| RSS | 同上 | OS ページキャッシュ含む使用量 |
| 状態遷移 (NORMAL/PRESSURE/CRITICAL) | nproxy debug log の POLICY 行 | policy が走ったか |
| pipe stall | runner 側 5秒tick が30秒間進まない判定 | 実 stall 検出 |
| スループット | finalBytes / durationMs | ランタイムの素の実力 |

---

## 3. 結果（実機）

実行環境: Windows 11 (10.0.22631 x64) / Node.js v12.9.1 / SSD

| サイズ | 所要時間 | 平均ThroughPut | heap min/max/avg (MB) | RSS min/max/avg (MB) | 状態 | stall | passed |
|---:|---:|---:|---:|---:|:---:|:---:|:---:|
| 1 GB | 1.78 s | 576 MB/s | 2.2 / 2.5 / 2.3 | 28 / 58.7 / 40.3 | NORMAL のみ | なし | ✅ |
| 5 GB | 6.11 s | 837 MB/s | 2.3 / 2.7 / 2.5 | 29.4 / 100.9 / 68.1 | NORMAL のみ | なし | ✅ |
| 10 GB | 9.87 s | 1037 MB/s | 2.4 / 2.7 / 2.6 | 34.4 / 102.4 / 82.7 | NORMAL のみ | なし | ✅ |

---

## 4. 数字が意味すること

### 4.1 heap がサイズに比例して増えていない

10倍データ流しても heap は **2.5 MB → 2.6 MB** という誤差レベル。
これはつまり:

- **chunk が JS ヒープにほぼ滞在していない**
- BackingStore は libuv が確保 → V8 へ external として渡す → 即書き出し → 即解放
- 観測は size と ts のメタオブジェクトのみ（数十バイト × ring の最大1024件 = 数十 KB）

### 4.2 RSS がサイズではなく時間に追従している

RSS は 28 → 100MB あたりに張り付く。これは **OS のパイプバッファと
ファイル書き込みキャッシュ** が見えているもの。
GC ではなく **OS が管理しているメモリ**であり、設計上の問題ではない。

### 4.3 スループットが大規模ほど上がる

| サイズ | MB/s |
|---:|---:|
| 1 GB | 576 |
| 5 GB | 837 |
| 10 GB | 1037 |

これは「初期化コスト（spawn / require / pipe 接続）が定数で
データが大きくなるほど償却される」ことを示す。
**ランタイム自体は速度の天井になっていない**。

### 4.4 状態遷移しなかった意味

PRESSURE 閾値 80MB に対し、heap は終始 2〜3MB。
**policy が走らなかったのは "走らせる必要がなかった" から**で、これは前段の
「chunk非保持」がそれだけで十分機能したことを意味する。

---

## 5. では Node の天井はどこにあるのか

実機で Node を「破壊」できなかった以上、天井は数字の延長線上で推定する。

| 想定上限 | 値 | 根拠 |
|---|---|---|
| 単一 string サイズ | 約 2GB (V8 string max length) | ただし本設計では string 化しない |
| `kIoMaxLength` (readFile) | 約 2GB | createReadStream を使えば回避 |
| RSS (Windows プロセス) | 8 TB (x64 仮想) | 物理メモリと OS 設定次第 |
| pipe バッファ (Windows) | 64 KB / pipe（OS 既定） | C++ pipe で backpressure が回るため非問題 |
| fd 数 | プロセス当たり 65535 (Win) | 本件は3本のみ |

つまり結論として:

> **「設計上、Node が天井になる場面はもう設計の外」**。
> 例えば 100 GB のリアルタイム ETL や、数百本の pipe を同時に張る
> マルチプレキサ、splice/sendfile によるゼロコピー要件など。

---

## 6. 「Node を破壊するシナリオ」逆算

意図的に nproxy を壊そうとすると、以下のいずれかが必要。

1. **設計原則を破る**: stream の data イベントで chunk を保持し始める
   → 即 OOM。これは設計の問題で、ランタイムの問題ではない。
2. **ヒープを直接膨らませる**: 子側で `Buffer.alloc(2*1024*1024*1024)` を 4 本確保
   → child が落ちる。nproxy 側は無傷（流量経由でしか見えない）。
3. **同時 spawn 数を 100 本**: nproxy を 100 個動かす
   → OS の fd / pipe バッファ枯渇。ランタイム差し替えが意味を持ち始める。
4. **ファイルではなくネットワークへ吐く**: latency と TCP window が支配項に
   → backpressure の効き方が変わる。Node でも対応可能だが
   ゼロコピー (sendfile) が欲しくなる領域。

---

## 7. 結論

| 問い | 答え |
|---|---|
| 設計は Node 上で持つか | **持つ。10GB まで heap フラット** |
| Node の限界を計測できたか | **計測範囲では出なかった**。設計が先に Node の長所を引き出している |
| Rust に進む正当性は | **明確にある**。ただし「Node では足りない」ではなく **「同じ思想を下層で再証明し、機能を拡張するため」** |

## 8. 次フェーズへの引き継ぎ

- **設計原則3点**は普遍。言語に依存しない:
  1. chunk 非保持
  2. backpressure 委譲
  3. policy は副作用縮退のみ
- Node PoC のメトリクスを **基準値 (baseline)** として残す:
  - 5GB / 6.1s / 837 MB/s / heap 2.5MB
- Rust MVP は **同じテストスクリプトで比較可能** にする
  （`run_limit_test.js` の APP 引数を Rust nproxy に差し替えるだけ）
