## 1. 今日からできる3つのこと

以下の3つは今日中に実施でき、効果が高く、コストもかかりません。

### 1-1. 管理用 PC の見直し

サーバーは Linux ベースのレンタルサーバーです。管理用 PC も **Linux (Ubuntu デスクトップ)** を推奨します。

**理由:**
- SSH がネイティブで使える（追加ソフト不要）
- サーバーと同じ OS なので、ファイルの改行コード問題がない
- wp-cli 等の WordPress 管理ツールがそのまま動作
- マルウェアの標的になりにくい（Windows に比べて攻撃対象が少ない）

**Ubuntu デスクトップなら Windows とほぼ同じ操作感:**
- マウス操作でファイル管理、ブラウザ、メールが使える
- 初心者向けのディストリビューション (Ubuntu) なら日本語サポートも充実
- 無料

**Linux のもう一つのメリット — CLI ツールと AI の活用:**
- ログ解析、ファイル検索、テキスト処理など CLI ツールが豊富 (grep, awk, sed, jq 等)
- AI アシスタント (Claude, ChatGPT 等) が CLI 操作をサポート。コマンドを聞けば教えてくれる
- 自動化スクリプト (Bash, Python) で定型作業を自動化可能
- Windows では面倒なサーバー管理作業が、Linux ではネイティブで完結
- MCP Server のような外部ツール連携が不要 — Linux には `grep`, `awk`, `sed`, `curl`, `jq` 等のCLIツールが標準で揃っている。Windows では PowerShell のみで CLI ツールが限られているため MCP Server を使う場面が多いが、Linux ではその必要がない

**現在の PC が Windows 10 の場合:**
- Windows 10 は **2025年10月に無料のセキュリティパッチ提供が終了済み**です（有料の延長セキュリティアップデート (ESU) はあるが個人向けではない）。早急に OS を移行してください
- Windows 11 への移行も可能ですが、サーバー管理が主な用途なら Linux に移行する方が合理的です

> **💡 コスト:** Ubuntu デスクトップは**無料**。Windows 11 へのアップグレードも**無料**（PC スペック要確認）。

### 1-2. パスワードマネージャーを導入する

サーバー、WordPress、FTP の認証情報を安全に管理します。

- **推奨ツール:** Bitwarden（無料）または 1Password（月額約500円）
- すべてのパスワードを**使い回さない**（サイトごとに異なるパスワードを設定）
- 推測されにくいパスワードを自動生成（例: `Kj#9mP$vL2@nQ5x`）

### 1-3. 2段階認証 (2FA) を有効にする

パスワードが漏洩しても、2段階認証があれば不正アクセスを防げます。

- WordPress 管理画面に 2FA を設定（プラグイン: **Two Factor Authentication**）
- サーバーパネル（cPanel等）に 2FA を設定
- Google アカウントに 2FA を設定（Google Drive を使用している場合）

## 2. クラウドサービスの運用改善

> **⚠ 問題: 業務と個人の混在**
> クラウドサービスを個人用途と業務で区別していない場合、個人アカウントが乗っ取られるとサーバーの認証情報が漏洩するリスクがあります。

### 2-1. 業務用アカウントの分離

| 項目 | 現状のリスク | 推奨対策 |
|------|------------|---------|
| Google Drive | 個人アカウントに認証情報が保存 | 業務用アカウントを新規作成、またはパスワードマネージャーに移行 |
| OneDrive | 個人アカウントと混在 | 業務用フォルダを分離、共有設定を「特定ユーザーのみ」に限定 |
| iCloud | Apple ID に認証情報が保存されるリスク | 業務情報は iCloud に保存しない、パスワードマネージャーに移行 |
| メール | 個人メールで業務連絡 | 業務用メールアドレスを設定 |

### 2-2. 認証情報の保管場所

サーバーの認証情報 (FTP、SSH、WordPress管理画面のパスワード) は、**以下の場所に保存しないでください:**

- ❌ メールの下書き
- ❌ Google Drive / OneDrive / iCloud のテキストファイル
- ❌ ブラウザのブックマーク
- ❌ デスクトップのメモ帳ファイル

**推奨:** パスワードマネージャー (Bitwarden等) に保存。暗号化されて安全です。

## 3. WordPress はなぜ狙われるのか

### 3-0. WordPress の脆弱性について

WordPress は世界で最も利用されている CMS (コンテンツ管理システム) です。その分、**攻撃者にとって最も攻撃しやすいターゲット**でもあります。

**なぜ WordPress が狙われるのか:**
- **シェアが高い:** 全ウェブサイトの約40%が WordPress を使用。攻撃者は1つの脆弱性で数百万サイトを攻撃できる
- **攻撃コストが低い:** WordPress の脆弱性情報はネットに公開されている。攻撃者はその情報をそのまま使える
- **プラグインの品質がまちまち:** 無料プラグインには脆弱性を含むものがある。開発者が放置したものも多い
- **更新が遅れがち:** WordPress 本体やプラグインの更新を放置すると、既知の脆弱性を突かれる

**WordPress の現実:**
- 攻撃者は「WordPress を使っているサイト」を自動でスキャンし、脆弱性を見つけて攻撃する
- 手動で対策しても、新しい脆弱性が毎日見つかる
- **「対策した」は「対策し続けなければならない」** という意味

**将来的な選択肢:**
- WordPress からの移行も検討に値する (静的サイト生成ツール: Hugo, Jekyll 等)
- ただし移行には技術的知識が必要。現時点では WordPress のセキュリティ対策を徹底する方が現実的

## 4. WordPress のセキュリティ対策

### 4-1. 更新管理

WordPress の脆弱性の多くは**未更新のプラグインやテーマ**から発生します。

- WordPress 本体を最新版に更新
- すべてのプラグインを最新版に更新
- 未使用のプラグインは**削除**（無効化ではなく削除）
- 未使用のテーマも**削除**
- 更新は**月1回以上**の頻度で確認

### 4-2. ログインセキュリティ

- **ログイン試行回数制限:** プラグイン「Limit Login Attempts Reloaded」を導入
- **XML-RPC の無効化:** ブルートフォース攻撃の入口になるため、`.htaccess` で無効化

```apache
<Files xmlrpc.php>
    Require all denied
</Files>
```

- **管理者アカウント名の変更:** 「admin」以外の名前に変更

### 3-3. ディレクトリ・ファイル名の変更

一度サーバーを乗っ取られた経験がある場合、**標準的な対策だけでは不十分です。** 攻撃者は WordPress のデフォルト構造を知っており、自動スクリプトで `/wp-content/uploads/` や `/wp-login.php` に直接アクセスしてきます。

**推奨:**
- `wp-content/` を別の名前に変更 (例: `assets/`, `data/` 等)
- `wp-login.php` を変更 (プラグイン: **WPS Hide Login**)
- `wp-includes/` を変更 (高度な設定が必要)

**注意点:**
- プラグインによっては `wp-content` をハードコードしているものがあり、動作しなくなる可能性がある
- テーマの CSS/JS パスも変わるため、表示崩れが起きる可能性がある
- 変更後は**必ずテスト環境で動作確認**してから本番に反映する
- WordPress のコア更新で戻る可能性があるため、更新後に再設定が必要

**最重要:** これは標準的な対策ではありません。攻撃者に「このサイトは普通の WordPress ではない」と思わせる効果があります。ただし、管理できる範囲内で実施してください。

### 3-4. 独自のセキュリティ対策

ネットに公開されている「おすすめセキュリティ対策」は攻撃者も見ています。**独自の対策を講じることが重要です。**

具体例:
- ログインページの URL を独自に変更 (プラグインでは公開情報になってしまう)
- アクセスログを独自の場所に保存
- 独自の IP ブロックリストを管理
- 定期的にログを手動で確認 (自動化に頼りすぎない)
- サーバーの SSH ポートを変更 (デフォルトの 22 番から変更)

**原則: 「何をやっているか」を公開しない。** 攻撃者は公開情報を手がかりに攻撃してきます。

### 3-8. 脆弱性の日常的なチェック — サイト管理の一部

攻撃を受けてから対策するのは論外です。**堅牢なサイトは独自の対策を講じつつ、日々の脆弱性情報を把握しています。**

WordPress は毎日のように脆弱性が報告されています。これは WordPress が悪いのではなく、利用者数が多いためセキュリティ研究者の注目を集めている結果です。

**日常的なチェック項目:**
- WordPress 本体・プラグイン・テーマの脆弱性情報を毎日確認
- 使用しているプラグインの開発状況を定期的に確認 (放置されていないか)
- WordPress の公式セキュリティブログを購読
- 日本語の脆弱性情報サイト (WPVulnerability 等) をブックマーク

**推奨情報源:**
- WordPress 公式: https://wordpress.org/news/
- WPScan Vulnerability Database: https://wpscan.com/vulnerabilities
- JVN (日本): https://jvn.jp/

**サイト管理は「コンテンツを更新する」だけではありません。** セキュリティ情報を確認し、脆弱性が見つかったら即座に対応する。これがサイト管理を続ける上で避けられない業務です。

### 3-5. データベースの強化

WordPress はデフォルトでデータベース名が `wordpress`、テーブル接頭辞が `wp_` です。攻撃者はこの構造を知っています。

**推奨:**
- データベース名を変更 (例: `wordpress` → `myapp_db_2026` 等)
- テーブル接頭辞を変更 (`wp_` → `app_` 等。インストール時に設定可能)
- データベースのユーザー名を `root` や `wordpress` 以外にする

**wp-config.php の認証情報について:**
WordPress のデータベース接続情報は `wp-config.php` に**平文 (暗号化されていない状態)** で記載されています。このファイルが漏洩するとデータベースの認証情報が丸ごと漏洩します。

```php
// wp-config.php の中身 (例)
define('DB_NAME', 'wordpress');      // データベース名
define('DB_USER', 'root');           // ユーザー名
define('DB_PASSWORD', 'password123'); // パスワード (平文)
```

**対策:**
- `wp-config.php` の権限を **400** (所有者のみ読み取り可) に設定
- データベースのパスワードを推測されにくいものに変更
- 可能であれば `wp-config.php` をドキュメントルートの外に移動（共有レンタルサーバーでは移動できない場合が多いため、その場合は権限設定 (400) で対応する）

### 3-6. ログの観察 — 最も有効な対策

自動化されたセキュリティツールに頼るだけでなく、**人間がログを定期的に目視で確認する**ことが最も有効です。

**なぜログの観察が重要か:**
- 自動化ツールは既知の攻撃パターンしか検出できない
- 新しい攻撃手法や独自のパターンは人間でしか気づけない
- 「いつもと違う」異変に気づくのは人間の感覚

**確認すべきログ:**
- **アクセスログ** (`access.log`): 異常な IP、異常なリクエストパターン、存在しないページへのアクセス
- **エラーログ** (`error.log`): PHP エラー、ファイルアクセスエラー、データベース接続エラー
- **ログイン履歴**: 異常な時間帯のログイン、存在しないユーザー名でのログイン試行
- **ファイル変更履歴**: 予期しないファイルの追加・変更・削除

**具体的な確認方法:**
- **毎日:** ログイン履歴を確認 (5分)
- **毎週:** アクセスログをざっと確認 (15分)
- **毎月:** ファイル変更履歴を確認 (30分)

**異常を見抜くポイント:**
- 同一 IP から短時間に大量のリクエスト → ボットの可能性
- 存在しないファイルへのアクセス (`/wp-config.php.bak` 等) → ファイル漏洩の探索
- 異常な時間帯 (深夜2〜5時) のアクセス → 攻撃の可能性
- 通常とは異なる User-Agent → ボットの可能性

**泥臭い作業ですが、これが最も確実です。** 自動化ツールに頼りすぎず、人間の目で確認してください。

### 3-7. ログ分析の効率化 — AI エージェントの活用

ログの手動確認は重要ですが、**AI エージェントを活用すれば分析を最大限に効率化できます。**

**推奨方法:**
- SSH でサーバーに接続し、AI エージェント (Claude, ChatGPT等) にログ分析を依頼
- 「このアクセスログから異常なパターンを見つけ出して」等の指示で AI が分析
- GPU マシンがあるなら、**無料のローカル AI (Ollama, llama.cpp等)** で十分。サーバーのログを外部に送る必要がない
- ローカル AI なら機密性の高いログデータも安全に処理できる

**Linux 系のサーバー管理者にとって、AI を使ったログ分析は鉄則です。** 手動でログを確認する時間がない場合でも、AI に任せれば短時間で異常を検出できます。

### 3-8. 緊急時対応 — リアルタイム対応できる業者の確保

セキュリティ対策を万全にしても、攻撃を100%防ぐことはできません。**緊急時にリアルタイムで対応できる業者を確保しておくことが必須です。**

**求める要件:**
- 24時間365日の対応 (攻撃は深夜・休日にも起きる)
- WordPress のセキュリティに詳しい技術者が在籍
- サーバー侵入時の復旧対応が可能
- フォレンジック (攻撃の原因調査) に対応できる

**事前に契約しておくこと:**
- 緊急連絡先を把握しておく
- 対応範囲と費用を事前に確認しておく
- 侵入された場合の復旧手順を事前に共有しておく

### 3-7. ファイル権限の設定

| ファイル | 推奨権限 | 理由 |
|---------|---------|------|
| wp-config.php | 400 | 所有者のみ読み取り可。最も重要な設定ファイル |
| .htaccess | 444 | 読み取りのみ。変更を防止 |
| wp-content/ | 755 | プラグイン・テーマ・アップロード用 |

## 4. 検証環境の構築

> **⚠ 問題: 検証環境がない**
> 本番環境で直接作業していると、設定ミスやプラグイン更新でサイトが即座にダウンするリスクがあります。

### 4-1. ローカル環境の構築

PC 上にテスト環境を作り、本番反映前にテストを行います。

- **推奨ツール:** Local by Flywheel（無料）— WordPress のローカル環境を簡単に構築
- 設定変更やプラグイン更新はまずローカル環境でテスト
- 問題がなければ本番環境に反映

### 4-2. ステージング環境の構築

本番サーバーにテスト用のサブドメインを作り、本番と同じ環境でテストを行います。

- 例: `staging.example.com` を作成
- 本番データをコピーしてテスト
- テスト完了後に本番に反映

## 5. バックアップ体制

### 5-1. バックアップの基本方針

| 項目 | 推奨 |
|------|------|
| 頻度 | 日次（毎日自動バックアップ） |
| 対象 | WordPress ファイル + データベース |
| 保存先 | サーバーとは異なる場所（ローカル PC + クラウド） |
| 検証 | 月1回、リストアテストを実施 |

### 5-2. バックアップツール

- **プラグイン:** UpdraftPlus（無料版あり）— 自動バックアップ + クラウド保存
- **手動:** FTP でファイルをダウンロード + phpMyAdmin でデータベースをエクスポート

> **💡 重要なポイント**
> バックアップは「取る」だけでは不十分です。**定期的にリストアテストを行い、復元できることを確認**してください。バックアップが復元できないと、被害発生時にサイトを復旧できません。

## 6. 攻撃の段階と対応


サイバー攻撃は突然起きるのではありません。攻撃者はまず**偵察**を行い、脆弱性を見つけてから攻撃を仕掛けます。**偵察段階で検知し、対策を講じることが最も重要です。**

### 攻撃の段階

```
偵察 → 脆弱性の発見 → 侵入 → 権限昇格 → データ窃取/改ざん → 被害発覚
```

**「攻撃を受けた → アウト」ではなく、「偵察の時点で何かしらの対策をすべき」です。**

### 6-1. 偵察段階の検知 — ここで防ぐ

攻撃者はまずサイトの情報を収集します。この段階で異常に気づけば、攻撃を未然に防げます。

**偵察の兆候:**
- 存在しないページへのアクセス (`/wp-config.php.bak`, `/backup/`, `/.git/` 等)
- 異常な User-Agent (空、既知のボットUA、スキャンツールのUA)
- 同一 IP からの短時間大量リクエスト (ポートスキャン、ディレクトリスキャン)
- 異常な時間帯 (深夜2〜5時) のアクセス
- ログインページへの異常なアクセスパターン

**対応:**
- 上記の兆候を発見したら、**即座に該当 IP をブロック**
- ログを詳細に確認し、攻撃者の行動パターンを分析
- 同じ攻撃パターンが再来しないか監視を強化

### 6-2. 脆弱性発見段階 — ここで対策

攻撃者が脆弱性を発見した段階。まだ侵入はされていない。

**兆候:**
- 特定のプラグインやファイルへの異常なアクセス集中
- SQL インジェクション試行の痕跡 (`' OR 1=1`, `UNION SELECT` 等)
- ファイルアップロードの試行

**対応:**
- 該当するプラグイン/テーマを即座に更新 or 無効化
- WAF ルールで攻撃パターンをブロック
- サーバーのログを監視強化

### 6-3. 侵入された場合 — 最悪のケース

偵察を見逃し、脆弱性を突かれて侵入された場合。

**対応手順:**
1. **すぐにサーバーにログインし、不審なファイルを確認**
2. FTP/SSH のパスワードを**直ちに変更**
3. バックアップからサイトを復旧
4. WordPress 本体・プラグイン・テーマをすべて最新に更新
5. 不審なプラグイン・テーマを削除
6. 管理者アカウントのパスワードを変更
7. データベースのパスワードを変更
8. 攻撃の原因を調査し、同じ脆弱性が残っていないか確認

### 6-2. 対応時

1. バックアップからサイトを復旧
2. WordPress 本体・プラグイン・テーマをすべて最新に更新
3. 不審なプラグイン・テーマを削除
4. 管理者アカウントのパスワードを変更
5. データベースのパスワードを変更

### 6-3. 事後対応

1. 攻撃の原因を調査（どの脆弱性を突かれたか）
2. 同じ脆弱性が残っていないか確認
3. セキュリティ強化策を実施
4. 必要に応じて専門業者に相談

## 7. 無料でできる対策のまとめ



| 対策 | コスト | 工数 | 効果 |
|------|--------|------|------|

## 9. セキュリティ情報の取り扱いについて


### 9-1. ネットの情報を鵜呑みにしない

インターネットには「このセキュリティ対策が効果的」「このプラグインを入れれば安全」といった情報が溢れていますが、**攻撃者も同じ情報を読んでいます。**

- 「このプラグインで対策してます」→ 攻撃者はそのプラグインの脆弱性を調べる
- 「この設定で防いでます」→ 攻撃者はその設定の回避方法を探す
- 「この WAF で保護してます」→ 攻撃者はその WAF のバイパス方法を知っている

**対策を公開することは、攻撃者に「攻略法」を教えることと同じです。**

### 9-2. 堅牢なサイトは非公開

実際にセキュリティが高いサイトは、自社のセキュリティ対策を公開しません。

- どのような対策を講じているかは**社外秘**
- 使用しているツールや設定は**公開しない**
- 攻撃者は公開情報を手がかりに攻撃してくる

**推奨:**
- 自社のセキュリティ対策をブログや SNS で公開しない
- 「どの対策をしているか」を外部に漏らさない
- 攻撃者に「このサイトは簡単に攻撃できない」と思わせることが最大の防御

### 9-3. 本文書の取り扱い

この文書には具体的な対策方法が記載されていますが、**外部には公開しないでください。** 攻撃者に攻略法を教えることになります。

## 10. まとめ


セキュリティ対策の基本は**「やれることから始める」**です。すべてを一度に実施する必要はありません。

1. **今日:** Windows Update、パスワードマネージャー、2FA
2. **今週:** クラウドの業務・個人分離、WordPress プラグイン更新
3. **今月:** 検証環境構築、バックアップ体制
4. **継続:** 定期的な更新確認と見直し

> **💡 最も重要なこと**
> セキュリティ対策は「完璧」を目指すのではなく、「攻撃者のコストを上げる」ことが目的です。少しの対策で、多くの攻撃を防げます。
