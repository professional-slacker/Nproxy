セキュリティ改善アドバイス

ウェブサイト保守担当者向け — 2026年5月27日

⚠ 緊急度: 高

過去1年間にマルウェアによるサーバー乗っ取り、Form爆撃の被害を受けています。この文書は、今後の被害を防止するための具体的な対策をまとめたものです。

1. 今日からできる3つのこと

以下の3つは、今日中に実施でき、効果が高く、コストもかかりません。

1-1. Windows Update を最新にする

お使いの PC が Windows 10 の場合、2025年10月にサポートが終了します。それ以降はセキュリティパッチが提供されません。

  • 「設定」→「Windows Update」→「更新プログラムのチェック」を実行
  • 2025年10月までに Windows 11 への移行を計画
  • Windows 11 に非対応の PC の場合は、Linux (Ubuntu等) の導入を検討
💡 コスト

Windows 11 へのアップグレードは無料です（PC のスペック要確認）。

1-2. パスワードマネージャーを導入する

サーバー、WordPress、FTP の認証情報を安全に管理します。

  • 推奨ツール: Bitwarden（無料）または 1Password（月額約500円）
  • すべてのパスワードを使い回さない（サイトごとに異なるパスワードを設定）
  • 推測されにくいパスワードを自動生成（例: Kj#9mP$vL2@nQ5x）

1-3. 2段階認証 (2FA) を有効にする

パスワードが漏洩しても、2段階認証があれば不正アクセスを防げます。

  • WordPress 管理画面に 2FA を設定（プラグイン: Two Factor Authentication）
  • サーバーパネル（cPanel等）に 2FA を設定
  • Google アカウントに 2FA を設定（Google Drive を使用している場合）

2. クラウドサービスの運用改善

⚠ 問題: 業務と個人の混在

Google Drive などのクラウドサービスを個人用途と業務で区別していない場合、個人アカウントが乗っ取られるとサーバーの認証情報が漏洩するリスクがあります。

2-1. 業務用アカウントの分離

項目 現状のリスク 推奨対策
Google Drive 個人アカウントに認証情報が保存 業務用アカウントを新規作成、またはパスワードマネージャーに移行
メール 個人メールで業務連絡 業務用メールアドレスを設定
共有設定 誤操作で認証情報が公開されるリスク 共有範囲を「特定ユーザーのみ」に限定

2-2. 認証情報の保管場所

サーバーの認証情報 (FTP、SSH、WordPress管理画面のパスワード) は、以下の場所に保存しないでください:

  • ❌ メールの下書き
  • ❌ Google Drive のテキストファイル
  • ❌ ブラウザのブックマーク
  • ❌ デスクトップのメモ帳ファイル

推奨: パスワードマネージャー (Bitwarden等) に保存。暗号化されて安全です。

3. WordPress のセキュリティ対策

3-1. 更新管理

WordPress の脆弱性の多くは、未更新のプラグインやテーマから発生します。

  • WordPress 本体を最新版に更新
  • すべてのプラグインを最新版に更新
  • 未使用のプラグインは削除（無効化ではなく削除）
  • 未使用のテーマも削除
  • 更新は月1回以上の頻度で確認

3-2. ログインセキュリティ

  • ログイン試行回数制限: プラグイン「Limit Login Attempts Reloaded」を導入
  • XML-RPC の無効化: ブルートフォース攻撃の入口になるため、.htaccess で無効化 
    <Files xmlrpc.php>
    Require all denied
</Files>
  • 管理者アカウント名の変更: 「admin」以外の名前に変更

3-3. ファイル権限の設定

ファイル 推奨権限 理由
wp-config.php 400 所有者のみ読み取り可。最も重要な設定ファイル
.htaccess 444 読み取りのみ。変更を防止
wp-content/ 755 プラグイン・テーマ・アップロード用

4. 検証環境の構築

⚠ 問題: 検証環境がない

本番環境で直接作業していると、設定ミスやプラグイン更新でサイトが即座にダウンするリスクがあります。

4-1. ローカル環境の構築

PC 上にテスト環境を作り、本番反映前にテストを行います。

  • 推奨ツール: Local by Flywheel（無料）— WordPress のローカル環境を簡単に構築
  • 設定変更やプラグイン更新は、まずローカル環境でテスト
  • 問題がなければ本番環境に反映

4-2. ステージング環境の構築

本番サーバーにテスト用のサブドメインを作り、本番と同じ環境でテストを行います。

  • 例: staging.example.com を作成
  • 本番データをコピーしてテスト
  • テスト完了後に本番に反映

5. バックアップ体制

5-1. バックアップの基本方針

項目 推奨
頻度 日次（毎日自動バックアップ）
対象 WordPress ファイル + データベース
保存先 サーバーとは異なる場所（ローカル PC + クラウド）
検証 月1回、リストアテストを実施

5-2. バックアップツール

  • プラグイン: UpdraftPlus（無料版あり）— 自動バックアップ + クラウド保存
  • 手動: FTP でファイルをダウンロード + phpMyAdmin でデータベースをエクスポート
💡 重要なポイント

バックアップは「取る」だけでは不十分です。定期的にリストアテストを行い、復元できることを確認してください。バックアップが復元できないと、被害発生時にサイトを復旧できません。

6. Form爆撃への対策

6-1. お問い合わせフォームの保護

  • reCAPTCHA v3 の導入: ユーザーに負荷なし（バックグラウンドで判定）。無料
  • Honeypot フィールドの追加: 人間には見えないフィールドを設置し、ボットが入力したら破棄
  • 送信レート制限: 同一 IP からの連続送信を制限

6-2. 送信先の分離

フォーム送信先のメールアドレスを業務用に限定。個人メールアドレスへの送信は避ける。

7. インシデント対応手順

万一、攻撃を受けた場合の手順を事前に決めておきます。

7-1. 発見時

  1. サイトの異常を発見（表示がおかしい、不正なページが追加されている等）
  2. すぐにサーバーにログインし、不審なファイルを確認
  3. FTP/SSH のパスワードを直ちに変更

7-2. 対応時

  1. バックアップからサイトを復旧
  2. WordPress 本体・プラグイン・テーマをすべて最新に更新
  3. 不審なプラグイン・テーマを削除
  4. 管理者アカウントのパスワードを変更
  5. データベースのパスワードを変更

7-3. 事後対応

  1. 攻撃の原因を調査（どの脆弱性を突かれたか）
  2. 同じ脆弱性が残っていないか確認
  3. セキュリティ強化策を実施
  4. 必要に応じて専門業者に相談

9. 定期的な見直し

セキュリティ対策は「一度やったら終わり」ではありません。定期的な見直しが必要です。

頻度 確認項目
毎週 WordPress の更新確認、ログイン履歴の確認
毎月 プラグインの更新、バックアップのリストアテスト
四半期 パスワードの見直し、不要なプラグイン/アカウントの削除
年1回 セキュリティ全体の見直し、OS のアップデート確認

10. まとめ

セキュリティ対策の基本は「やれることから始める」です。すべてを一度に実施する必要はありません。

  1. 今日: Windows Update、パスワードマネージャー、2FA
  2. 今週: クラウドの業務・個人分離、WordPress プラグイン更新
  3. 今月: 検証環境構築、バックアップ体制、reCAPTCHA 導入
  4. 継続: 定期的な更新確認と見直し
💡 最も重要なこと

セキュリティ対策は「完璧」を目指すのではなく、「攻撃者のコストを上げる」ことが目的です。少しの対策で、多くの攻撃を防げます。
