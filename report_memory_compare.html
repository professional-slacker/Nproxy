nproxy Rust vs Node メモリ比較

Rust (tokio, Linux) Node.js (V8, Windows)

比較基準: Node版 report.html の実測値を正として、PCスペック差を考慮してメモリに焦点

測定条件の差異について
Node版は Windows 11 (Intel不明) + Node.js v12.9.1、Rust版は Debian Linux (Intel) + tokio 1.x で測定。 OS のメモリ管理方式が異なる（Windows: working set / Linux: RSS + page cache）ため、RSS の絶対値比較は参考程度とし、 プロセス固有の管理メモリ（Node: heapUsed / Rust: VmData）を主要指標とする。

5. 考察

5.1 Rust の優位点

  • ベースRSSが 8MB と圧倒的に軽量: Node の 19MB に対して 58% 削減。VM がないため起動時から低い。
  • RSS がデータサイズと無関係に完全フラット: Node は OS pipe cache 込みで RSS が 34→102MB と変動するが、 Rust は 8.3MB で張り付く（カーネル pipe buffer は nproxy の RSS に計上されない）。
  • 管理メモリの予測可能性: VmData 20MB の内訳はコード(4MB) + スタック(1MB) + 固定アロケーション。 Node の V8 GC のように突発的なメモリ回収が起きない。
  • idle と active でRSS差がほぼゼロ: Node は idle 19MB → 最大 102MB（5.4倍）。 Rust には V8 のようなヒープ拡張がない。

5.2 Node の優位点

  • heapUsed 2.7MB は V8 ヒープ内部の「純粋なアロケーション量」であり、実質的には Rust の VM なし環境より低い。
  • text decode 実装が動作中: Rust MVP は未実装だが、Node は 10GB string decode でも heap 19MB 以下を維持。
  • 1スレッド動作: アイドル時のスレッド数が 1 に対して Rust は 11〜12（tokio worker pool）。
  • メモリポリシー連動の実績: transform → passthrough の自動縮退が動作確認済み。

Rust の最大の強みは「ベースRSS 8MB でデータ量に全く依存しない」点にある。 Node は V8 VM の起動コスト 19MB が常に乗るが、V8 ヒープ内の割当 (heapUsed) は 2.7MB と非常に効率的。 管理メモリの「実質値」は両者ほぼ同等だが、OS から見たフットプリント (RSS) では Rust が有利。 text decode の実装が完了すれば、Rust は Node 同等の heap 特性をより低い RSS ベースで実現できる。
